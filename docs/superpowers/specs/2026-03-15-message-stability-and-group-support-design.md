# feishu-claude-code 优化设计文档

**日期**: 2026-03-15
**目标**: 解决消息返回不稳定问题，添加群组支持

---

## 问题陈述

### 问题 1：消息返回不稳定
当前实现使用 patch API 逐步更新卡片内容，存在以下问题：
- 返回内容有时被截断，不完整
- 卡片更新卡顿或长时间无响应
- patch 失败导致消息显示错误或格式混乱

**根本原因**: 频繁的 patch 调用和网络不稳定性导致更新失败或丢失。

### 问题 2：缺少群组支持
当前只支持私聊，不支持群组交互：
- 无法在群组中使用机器人
- 群组和私聊共享同一个 session，无法隔离

**需求**: 支持在不同群组中独立运行，每个群组有独立的 session。

---

## 解决方案

### 方案 1：消息稳定性 - 一次性发送

**设计**
- 移除 patch 流式更新逻辑
- 发送占位卡片"⏳ 思考中..."后，等待 Claude 完全回复
- 回复完成后，一次性发送完整卡片（或用单次 patch 替换）
- 工具调用进度仍在占位卡片顶部实时显示

**流程**
```
1. 用户发送消息
2. 发送占位卡片 "⏳ 思考中..."，获取 message_id
3. 调用 Claude CLI，积累完整回复
4. Claude 回复完成
5. 用单次 patch 替换占位卡片为完整内容
6. 保存 session
```

**优点**
- 100% 消息完整性，无截断
- 无 patch 失败重试逻辑，代码简化
- 飞书 API 调用更稳定

**缺点**
- 长回复需要等待（通常 5-30 秒）
- 失去实时打字机效果

**实现细节**
- `claude_runner.py`: 积累完整回复,不再逐块回调
- `main.py`: 移除 `STREAM_CHUNK_SIZE` 和频繁 patch 逻辑,改为一次性 patch 替换占位卡片
- `feishu_client.py`: 简化,仅保留单次 patch 用于替换占位卡片

**占位卡片替换策略**
- 采用单次 patch 替换占位卡片为完整内容（而非删除）
- 理由：保持消息历史完整，用户体验更好
- 这不违反"移除频繁 patch"的目标，因为只有一次 patch 调用
- 如果 patch 失败，保留占位卡片，用户知道在处理中

**超时和错误处理**
- 设置最大等待时间：120 秒
- 如果超时，发送错误提示："回复超时，请重试"
- 如果 Claude 返回错误，发送错误信息到卡片
- 如果占位卡片发送失败，重试 3 次后放弃

### 方案 2：群组支持

**设计**
- Session 存储从 `user_id` 改为 `(user_id, chat_id)` 组合键
- 私聊时 `chat_id = user_id`
- 群聊时 `chat_id = group_id`
- 群聊中发任何消息都响应（无需 @机器人）

**数据结构改造**

旧结构：
```json
{
  "user_123": {
    "current_session": {...},
    "sessions": [...]
  }
}
```

新结构：
```json
{
  "user_123": {
    "private": {
      "current_session": {...},
      "sessions": [...]
    },
    "group_456": {
      "current_session": {...},
      "sessions": [...]
    },
    "group_789": {
      "current_session": {...},
      "sessions": [...]
    }
  }
}
```

**消息处理流程**
1. 识别消息来源（私聊 vs 群聊）
2. 提取 `chat_id`
   - 私聊：`chat_id = open_id`
   - 群聊：`chat_id = chat_id`（飞书 SDK 提供）
3. 所有 session 操作使用 `(user_id, chat_id)` 查询

**群聊特性**
- 群聊中发任何消息都响应（无需 @机器人）
- 每个群组独立的 session 历史
- 每个群组独立的模型配置（`/model`）
- 每个群组独立的权限模式（`/mode`）
- 每个群组独立的工作目录（`/cd`）
- `/resume` 在群组内只显示该群的历史

**配置作用域说明**
- 私聊配置：仅影响该用户的私聊 session
- 群聊配置：仅影响该群组的 session，不同群组独立
- 用户在私聊中的 `/model` 不影响其在群聊中的配置
- 用户在群聊中的 `/model` 不影响其在其他群聊或私聊中的配置

**私聊兼容性**
- 现有私聊 session 自动迁移到 `private` 子键
- 用户无感知，命令行为不变

## 权限和安全

### 群聊中的权限控制
- 所有用户都可以在群聊中发送消息触发机器人
- 所有用户都可以使用 `/model`、`/mode`、`/cd` 等命令
- 暂不实现群主/管理员权限限制（可作为后续功能）

### 安全考虑
- **成本风险**：群聊中每条消息都调用 Claude，可能导致成本增加
  - 缓解：监控 API 调用量，设置告警
- **隐私风险**：群聊中的代码/敏感信息会被 Claude 处理
  - 缓解：在环境提示中说明，建议用户谨慎
- **滥用风险**：恶意用户可能在群聊中大量调用机器人
  - 缓解：可添加速率限制（如每用户每分钟 5 条消息）

### 数据隐私
- 群聊消息和 session 数据存储在本地 JSON 文件中
- 不同群组的数据完全隔离
- 用户可以随时删除群组的 session 历史

## 数据迁移方案

### 迁移流程

**迁移前**
- 备份原 `~/.feishu-claude/sessions.json`
- 验证原数据完整性

**迁移脚本逻辑**（伪代码）
```python
def migrate_sessions():
    old_data = load_json("sessions.json")  # 旧格式
    new_data = {}

    for user_id, user_sessions in old_data.items():
        new_data[user_id] = {
            "private": user_sessions  # 所有旧数据移到 private 子键
        }

    # 验证迁移
    assert len(new_data) == len(old_data)
    for user_id in old_data:
        assert new_data[user_id]["private"] == old_data[user_id]

    # 备份原文件
    backup("sessions.json", "sessions.json.backup")

    # 写入新数据
    save_json("sessions.json", new_data)

    return True
```

**迁移后验证**
- 验证所有用户数据都在 `private` 子键中
- 验证数据完整性（session 数量、消息数量等）
- 测试 `/resume` 命令，确保旧 session 可恢复

**回滚方案**
- 如果迁移失败，恢复备份：`cp sessions.json.backup sessions.json`
- 如果迁移后发现问题，恢复备份并重新迁移

### 迁移期间的系统可用性
- 迁移前停止机器人（避免并发修改）
- 迁移通常 <1 秒（JSON 文件较小）
- 迁移后重启机器人

### 兼容性说明
- 迁移后，旧的直接查询 `user_id` 会失败
- 所有代码必须改为查询 `(user_id, chat_id)`
- 没有向后兼容的旧接口

## 存储优化

### 存储大小预期
- 每个 session：~5-50 KB（取决于消息数量）
- 每个用户：~100 KB - 1 MB（取决于群组数量和历史）
- 预期增长：每月 ~10-100 MB（取决于使用频率）

### 优化策略
- 暂不实现定期清理（可作为后续功能）
- 建议用户定期备份 `~/.feishu-claude/sessions.json`
- 如果文件过大（>100 MB），可手动删除旧 session

### 分布式存储
- 当前实现仅支持本地 JSON 存储
- 如需分布式存储，可改为 SQLite 或数据库（后续功能）

## 工具调用进度显示

### 设计决策
- 占位卡片发送后，不再实时更新进度
- 工具调用进度在回复完成后一起显示
- 理由：简化实现，避免频繁 patch 调用

### 用户体验
- 占位卡片显示"⏳ 思考中...",用户知道在处理
- 完成后显示完整回复,包括工具调用过程
